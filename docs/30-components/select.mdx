# Select

<BetaDocsBanner />

**Synonyme:** Dropdown, Dropdown-Liste, Auswahlliste, Auswahlfeld

**Beschreibung:** Die **Select**-Komponente ermöglicht die Auswahl einer oder mehrerer Optionen aus einer vordefinierten Liste.

Sie eignet sich für strukturierte Eingaben, bei denen ausschließlich bekannte Werte zulässig sind und keine freie Texteingabe erfolgen soll. Je nach Konfiguration unterstützt die Komponente sowohl die **Einfachauswahl** als auch die **Mehrfachauswahl.**

Die auswählbaren Optionen werden über die Property `_options` bereitgestellt und können optional mithilfe von **Optionsgruppen (Optgroups)** logisch strukturiert werden.

## Barrierefreiheit

Die Select-Komponente setzt konsequent auf native HTML-Funktionalitäten. Dadurch werden wesentliche Aspekte der Barrierefreiheit bereits durch Browser und assistive Technologien unterstützt. Eigene Accessibility-Implementierungen sind nur dort erforderlich, wo sie über den nativen Funktionsumfang hinausgehen.

- Browser stellen automatisch die korrekte Semantik sowie die erforderlichen ARIA-Rollen bereit.
- Tastaturbedienung, Fokusverwaltung und Screenreader-Unterstützung werden vollständig vom nativen HTML-Element übernommen.
- Die Beschriftung erfolgt über die Property `_label` und wird programmatisch mit dem Auswahlfeld verknüpft.
- Hinweise und Fehlermeldungen können über `_msg` bereitgestellt und von assistiven Technologien ausgegeben werden.
- Die Unterstützung der Mehrfachauswahl erfolgt über das native HTML-Attribut `multiple`. Dabei müssen Nutzende explizit über die Bedienung (Shift + Pfeiltasten) informiert werden.

**Hinweis zu `disabled`:**
Deaktivierte Optionen (`disabled`) können von Screenreadern weiterhin berücksichtigt oder angekündigt werden. Dadurch kann die Anzahl der angekündigten Optionen von der tatsächlich auswählbaren Anzahl abweichen. Das konkrete Verhalten ist abhängig von Browser und Screenreader.

### Konkrete Designentscheidungen

| Entscheidung | Begründung |
| ----------------- | ---------------------------- |
| Verwendung des nativen HTML-Elements `<select>` | Nutzt die standardisierte Semantik des Browsers und gewährleistet eine hohe Kompatibilität mit assistiven Technologien. |
| Verzicht auf eigene ARIA-Rollen | Das native `<select>` besitzt bereits die erforderliche Semantik. Zusätzliche Rollen wie `role="combobox"` oder `role="listbox"` würden keinen Mehrwert bieten und sollten nicht ergänzt werden. |
| Nutzung der nativen Browserimplementierung | Tastaturbedienung, Fokusverhalten und Screenreader-Ausgabe entsprechen den etablierten Webstandards und bleiben browserübergreifend konsistent. |
| Verwendung einer programmatisch verknüpften Beschriftung `(_label)` | Ermöglicht Screenreadern die eindeutige Zuordnung zwischen Beschriftung und Auswahlfeld. |
| Bereitstellung von Hinweisen und Fehlermeldungen über `_msg` | Unterstützt die barrierefreie Kommunikation von Validierungsfehlern und ergänzenden Informationen. |

## Verwendung

Die Select-Komponente eignet sich insbesondere für:
- die Auswahl einer oder mehrerer Optionen aus einer vordefinierten Menge an Werten,
- strukturierte Formulareingaben,
- standardisierte Datenerfassung,
- Einfach- und Mehrfachauswahlen.

Die Select-Komponente sollte **nicht** verwendet werden, wenn:
- freie Texteingaben erforderlich sind,
- eine Such- oder Filterfunktion benötigt wird,
- sehr große Datenmengen durchsucht werden müssen.

In diesen Fällen empfiehlt sich die Verwendung einer **Combobox**.

### Tastatursteuerung

Da die Select-Komponente auf dem nativen HTML-Element `<select>` basiert, entspricht die Tastaturbedienung dem Verhalten des jeweiligen Browsers und Betriebssystems.

| Taste                         | Funktion                                                 |
| ----------------------------- | -------------------------------------------------------- |
| `Tab`                         | Fokus auf die Komponente setzen                          |
| `Shift`+`Tab`                 | Zum vorherigen fokussierbaren Element wechseln           |
| `Pfeil-Tasten (oben / unten)` | Zwischen den Optionen navigieren (auch bei geschlossener Auswahlliste möglich) |
| `Enter`                       | Auswahlliste öffnen, Auswahl einer Option (schließt die Auswahlliste)         |
| `Leertaste`                   | Auswahlliste öffnen                                      |
| `Esc`                         | Auswahlliste schließen                                   |

#### Bei Mehrfachauswahl (`_multiple`)

| Taste                         | Funktion                                                 |
| ----------------------------- | -------------------------------------------------------- |
| `Pfeil-Tasten (oben / unten)` | Auswahl einer Option                                     |
| `Shift + Pfeil-Tasten`        | Auswahl aller hintereinanderliegenden Einträge           |
| `Strg + Pfeil-Tasten`         | Auswahl mehrerer einzelner Optionen                      |
| `Leertaste`                   | Auswahl einer weiteren Option                            |

### Best Practices / Empfehlungen

- Verwenden Sie die Select-Komponente ausschließlich für vordefinierte Auswahlmöglichkeiten.
- Nutzen Sie `_multiple` nur, wenn eine Mehrfachauswahl fachlich zwingend erforderlich ist. Eine Mehrfachauswahl kann für einige Nutzende schwer bedienbar sein, weshalb eine **Checkboxgruppe** häufig die verständlichere Alternative ist.
- Prüfen Sie bei umfangreichen Listen, ob eine Combobox die geeignetere Komponente ist.
- Strukturieren Sie umfangreiche Auswahllisten mithilfe von **Optionsgruppen (Optgroups)**.
- Stellen Sie die Optionen über die Property `_options` als Array mit `label` und `value` bereit.
- Verwenden Sie aussagekräftige und eindeutige Beschriftungen (`label`).
- Setzen Sie `_required`, wenn eine Auswahl zwingend erforderlich ist.
- Ergänzen Sie keine zusätzlichen ARIA-Attribute oder ARIA-Rollen, um das Verhalten nativer `<select>`- oder `<option>`-Elemente zu verändern. Die Komponente nutzt bewusst die native HTML-Semantik und profitiert dadurch von der standardisierten Unterstützung durch Browser und assistive Technologien.
- Treffen Sie keine automatische Vorauswahl personenbezogener Optionen (z. B. Anreden), sofern diese nicht fachlich begründet ist. Verwenden Sie stattdessen eine neutrale Option, damit Nutzende ihre Auswahl bewusst treffen können.


**Empfohlener Umgang mit `disabled`:**
- Sofern aus der Auswahlliste ein **sinnvoller, unstrittiger Default-Wert** hevorgeht, sollte dieser vorbelegt werden und es sollte **keine Leer-Option** geben. Dieses Vorgehen spart Zeit sowie Klicks und vermeidet Mehrdeutigkeiten.
- Falls aus der Auswahlliste **kein sinnvoller Default-Wert** hervorgeht oder die Wahl **rechtliche bzw. finanzielle Konsequenzen** hat, **kann** eine explizite, klar beschriftete Leer-Option verwendet werden (z. B. "Bitte auswählen").
- Grundsätzlich sollten Sie deaktivierte Optionen (`disabled`) vermeiden, sofern dies fachlich möglich ist. Sie können die Orientierung für Nutzende assistiver Technologien erschweren.

### Anwendungsfälle

- Anredeauswahl in Kontaktformularen
- Länder- oder Sprachauswahl
- Kategorieauswahl in Formularen
- Filterkriterien in Suchmasken
- Mehrfachauswahl bei Einstellungen oder Tags

### FAQ

**Wann sollte ich die Select-Komponente verwenden?**
Wenn Nutzende eine oder mehrere Optionen aus einer vordefinierten Liste auswählen sollen und keine freie Texteingabe erforderlich ist.

**Wann eignet sich eine Combobox besser?**
Wenn eine große Anzahl an Optionen zur Verfügung steht oder Nutzende gezielt nach Einträgen suchen sollen.

**Unterstützt die Komponente Optionsgruppen?**
Ja. Property `_options` unterstützt neben einzelnen Optionen auch **Optionsgruppen (Optgroups)**, um umfangreiche Listen logisch zu strukturieren.

**Unterstützt die Select-Komponente Mehrfachauswahl?**
Ja. Über die Property `_multiple` kann zwischen Einfach- und Mehrfachauswahl gewechselt werden.

**Warum verwendet die Komponente das native HTML-Element `<select>`?**
Die Verwendung des nativen HTML-Elements gewährleistet eine standardkonforme Semantik sowie eine browserseitige Unterstützung für Tastaturbedienung, Fokusverwaltung und Screenreader. Dadurch kann auf eigene Accessibility-Implementierungen und zusätzliche ARIA-Rollen verzichtet werden.
